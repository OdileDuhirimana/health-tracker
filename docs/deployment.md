# Deployment Guide (Render + Vercel)

This project is configured for:
- Backend on Render (`backend/`, NestJS API)
- Frontend on Vercel (`frontend/`, Next.js app)

## 1) Backend on Render

Use the repository blueprint in `render.yaml`.

Render creates:
- `vitals-careops-api` (web service)
- `vitals-careops-db` (PostgreSQL)
- `vitals-careops-cache` (Key Value / Redis-compatible, used for cross-instance rate limiting — see below)

Set these Render env vars on `vitals-careops-api`:
- `FRONTEND_URL`: your Vercel frontend URL (`https://<project>.vercel.app`)
- `CORS_ORIGIN`: comma-separated allowed origins (usually same as `FRONTEND_URL`)

Auto-provisioned by `render.yaml`:
- `DB_HOST`, `DB_PORT`, `DB_USERNAME`, `DB_PASSWORD`, `DB_DATABASE`
- `REDIS_URL` (from the `vitals-careops-cache` Key Value instance)
- `JWT_SECRET`

`vitals-careops-cache` has no free tier on Render, unlike the Postgres database above — this is a deliberate, disclosed cost tradeoff. The application does not require it to function: if `REDIS_URL` is absent, `RedisThrottlerStorage` (`backend/src/common/throttler/`) falls back to per-instance in-memory rate limiting automatically. Remove the `vitals-careops-cache` block and the `REDIS_URL` env var from `render.yaml` to opt out of the added cost, at the cost of rate limiting no longer being enforced globally if the service is ever scaled to more than one instance.

## 2) Frontend on Vercel

Deploy the `frontend` directory as a Vercel project.

Required Vercel env var:
- `NEXT_PUBLIC_API_URL`: Render backend URL, for example `https://vitals-careops-api.onrender.com`

`frontend/src/services/api-client.ts` fails fast (throws at request time with a descriptive error) if `NODE_ENV=production` and `NEXT_PUBLIC_API_URL` is unset, rather than silently falling back to `http://localhost:3001` — so a missing env var on a production Vercel build surfaces immediately instead of quietly targeting the wrong backend.

## 3) Where To Get Required Values

- `JWT_SECRET`: auto-generated by Render from `generateValue: true`
- `FRONTEND_URL` and `CORS_ORIGIN`: from your Vercel deployment domain
- `NEXT_PUBLIC_API_URL`: from your Render service URL
- `REDIS_URL`: auto-populated from the Render Key Value resource (`fromService` in `render.yaml`)
- DB variables: auto-populated from Render PostgreSQL resource

## 4) CI/CD: what's actually automated, and what isn't

**Continuous Integration** (`.github/workflows/ci.yml`) runs on every push/PR: backend lint, backend unit tests, backend e2e tests against a real ephemeral Postgres, a migration schema-check (see below), a backend production build, and equivalent lint/test/build jobs for the frontend. This is a genuine gate in the sense that it will fail loudly on a regression — but it is **not currently a hard merge gate**: GitHub branch protection rules (requiring these checks to pass before merge to `main`) are a repository setting, not something expressible in the workflow YAML itself, and have not been configured for this repository. To make CI an actually-enforced gate rather than an advisory one, enable "Require status checks to pass before merging" for `main` in the repository's branch protection settings and select the jobs in `ci.yml` as required checks.

**Continuous Deployment is platform-native auto-deploy-on-push, not an authored pipeline step.** Render and Vercel both watch the `main` branch directly and redeploy automatically on every push, independent of whether `ci.yml` passed. This means it is currently possible for a push that fails CI to still trigger a live deployment. This is an explicit, disclosed limitation, not an oversight:
- **Why it's this way:** both platforms' built-in git-push deploy is zero-maintenance and free at this project's scale; wiring an authored "deploy job gated on CI green" would mean either (a) disabling each platform's auto-deploy and replacing it with a `render.yaml`/Vercel CLI deploy step triggered from GitHub Actions after the `build` job succeeds, using a Render/Vercel API token stored as a repo secret, or (b) accepting the current gap.
- **Recommended hardening if this becomes a real production system** (not just a portfolio deployment): disable auto-deploy on both platforms, add a `deploy` job to `ci.yml` with `needs: [lint, unit-tests, e2e-tests, build]` that calls the Render/Vercel deploy APIs, and enable the branch-protection rule above so a failing CI run can never reach `main` in the first place.

**Migration safety in CI.** A dedicated `migration-check` job runs `npm run migration:run` against a completely empty, freshly-created Postgres instance (mirroring a real production restore-from-scratch or first deploy) and asserts the resulting schema actually contains the columns/constraints the application depends on. This exists because of a real bug this project shipped: the two pre-existing migrations could not run against a genuinely empty database at all (no migration had ever created the base tables — they only ever existed via `synchronize` in development), and separately, the `Dispensation` entity's `bucketType`/`bucketStart` columns and `uq_dispensation_bucket` constraint were never created by any migration. Both are now fixed (see `backend/src/migrations/1700000000000_initial_schema.ts` and `1740000000000_fix_dispensation_bucket_schema.ts`), and this CI job exists specifically so that class of bug can't silently reappear.

## 5) Rollback strategy

- **Application code:** both Render and Vercel retain previous successful deploys and support one-click rollback to a prior deploy from their respective dashboards. No additional configuration is required for this.
- **Database migrations:** each migration in `backend/src/migrations/` implements a `down()` method. To roll back the most recently applied migration, run `npm run migration:revert` (via the CLI DataSource in `backend/src/config/database.config.ts`) against the target database. This is a manual operation — there is no automated "deploy triggers rollback on failure" wiring, which is the honest state of a project at this scale rather than a claimed capability. Because Render's deploy (`npm run start:prod:migrate`) runs migrations automatically on every deploy, a bad migration should be fixed forward with a new corrective migration wherever possible rather than reverted, to avoid a window where the running application code and the database schema it expects are out of sync.
- **Environments:** there is currently no separate staging tier — only development (local) and production (Render/Vercel). A schema migration that needs verification against production-like data volume before reaching production has no dedicated environment to run in today; see the README's Future Improvements section.
